# Повышение исторических визитов до заказов

Раздел «Заказы» (`/admin/orders`) читает только сущность `Order`. Исторические
съёмки студии (до запуска раздела) существуют в базе только как `ClientVisit`
(восстановлены из Google-таблицы в прошлый раз) — у них никогда не было
`Order`. Этот скрипт создаёт `Order` + `ScheduleEvent` для каждого такого
визита, чтобы раздел «Заказы», карточка клиента, финансы и т.д. видели одну
и ту же историю из одного источника, без отдельной копии данных.

Источник — то, что УЖЕ есть в `ClientVisit` (see `src/lib/import/`), Google-
таблица повторно не запрашивается.

## Что переносится

- дата/время (`ClientVisit.startAt/endAt`, если распознаны при исходном импорте);
- зал, формат, длительность;
- стоимость (`grossAmount`) — считается полученной (`paymentStatus: PAID`),
  способ оплаты не придумывается (`paymentMethod: null`), если неизвестен;
- комментарий, с вырезанной фразой акции, если она распознана
  (`src/lib/promotion-model.ts` — тот же helper, что и везде на платформе);
- статус: `COMPLETED` (+ автоматически `isArchived`, если давность > 7 дней —
  та же логика, что и у обычных заказов, `isOrderReadyForArchive`), кроме
  визитов с признаком отмены/переноса в тексте — те остаются `BOOKED` для
  ручной проверки администратором.

## Что НЕ переносится автоматически

- **Абонементы.** Упоминание "абонемент" в старом комментарии — это только
  текст, не структурированная запись. Создавать `ClientSubscription`/
  `SubscriptionUsage` задним числом значило бы придумать финансовую историю,
  которой не было зафиксировано структурированно — решение владельца
  (2026-07-12): не делать этого автоматически. Такие визиты помечаются в
  dry-run для ручной проверки, комментарий остаётся как есть.
- **Способ оплаты** (картой/наличными/переводом) — исходная таблица его не
  фиксировала.
- **Материалы** (Яндекс.Диск/NAS) — исходная таблица их не содержала, заказ
  создаётся без ссылок ("Нет материалов" — как и для любого другого заказа
  без реальных ссылок).

## Команды

```bash
set -a && source .env.local && set +a

npx tsx scripts/promote-visits-to-orders/dry-run.ts     # ничего не меняет, только отчёт
npx tsx scripts/promote-visits-to-orders/apply.ts        # создаёт заказы, пишет манифест для отката
npx tsx scripts/promote-visits-to-orders/rollback.ts scripts/promote-visits-to-orders/backups/apply-....json
```

## Идемпотентность

`ClientVisit.orderId` (уникальное поле, добавлено этой доработкой) —
проставляется в момент создания заказа. `dry-run`/`apply` всегда запрашивают
визиты `WHERE orderId IS NULL` — уже повышенные визиты просто не попадают в
план при повторном запуске, дублей заказов создать невозможно.

## Откат

`apply.ts` пишет манифест в `scripts/promote-visits-to-orders/backups/apply-<дата>.json`
со списком реально созданных `{visitId, orderId, scheduleEventId}`.
`rollback.ts` с этим файлом удаляет ровно эти заказы/записи расписания и
возвращает `ClientVisit.orderId` в `null`.
