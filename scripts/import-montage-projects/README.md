# Импорт исторических проектов монтажа

Источник — Google-таблица "Монтаж" (вкладка с историей проектов, 68 строк на
момент написания, 07.10.2025–13.07.2026). Каждая строка становится
**самостоятельным** `MontageProject` (`orderId: null`, привязка к реальному
`Client`) — сама таблица не содержит ссылок на заказы/CRM, поэтому связать
исторические проекты с `Order` невозможно и не делается.

## Что переносится

- дата поступления (`Дата поступления заказа` → `sourceReceivedAt`);
- клиент — сопоставляется с существующим `Client` (см. "Сопоставление клиента" ниже);
- название/описание (`Столбец 4` → `title` — переименовано в понятное поле,
  ТЗ прямо просило не оставлять "Столбец 4");
- статус (`Сдан` → `DELIVERED`, `В работе` → `IN_PROGRESS`);
- ссылка на исходники (→ `sourceMaterialsUrl`);
- срок сдачи (→ `deadlineDate`, `deadlineType: FIXED_DATE`);
- суммы клиента/подрядчика (→ `clientAmount`/`editorAmount`; прибыль **не**
  сохраняется отдельным полем — считается `computeMontageProfit`, тот же
  helper, что и everywhere в разделе "Монтаж");
- исполнитель → отдельный `EditorProfile` (нормализация написания, см. ниже);
- количество итераций правок (→ `revisionsUsed`).

## Что НЕ переносится / не выдумывается

- **Фактическая дата сдачи (`deliveredAt`)** — в таблице есть только срок
  ("Срок сдачи"), не факт. Проставлять `deliveredAt = deadlineDate` означало
  бы придумать, что все исторические проекты сданы точно в срок — это
  никогда не проверялось, поэтому `deliveredAt` остаётся `null` даже для
  `DELIVERED`-проектов. Дедлайн в карточке виден, "сдано вовремя/с
  опозданием" для них просто не показывается (честно "неизвестно", а не
  придуманное "вовремя").
- **Даты оплаты (`clientPaidAt`/`editorPaidAt`)** — таблица их не содержит,
  остаются `null` даже для проектов, помеченных оплаченными.
- **Тип контента (`contentType`)** — не извлекается эвристикой из названия
  (риск ошибочной категоризации выше пользы); оставлен пустым, заполняется
  вручную при необходимости.
- **"Условия" и доп. исполнители через "/" или ","** — переносятся простым
  текстом в `internalComment` (структурировать под несуществующий в реальных
  данных случай не стали — см. ТЗ "не усложнять").

## Правило "неоплачено" — DATA-DRIVEN, не хардкод

Задача формулировала правило как "3 конкретных проекта" (Rebelgroup + 2×
Наталия/Нина). Это осознанно **не захардкожено** — вместо этого правило:

> `clientPaymentStatus = PENDING` и `editorPaymentStatus = PENDING`, если
> `Статус` в таблице на момент импорта = "В работе"; иначе `PAID`/`PAID`.

На момент написания скрипта в таблице **4** строки "В работе", а не 3 —
таблица живая и с момента постановки задачи в неё добавилась ещё одна строка
(ОБИТ, 13.07.2026). `dry-run.ts` показывает это расхождение явно и печатает
полную информацию по каждой такой строке — решение подтверждать/не
подтверждать импорт с этим списком остаётся за администратором.

## Сопоставление клиента

- **Точное совпадение** (без учёта регистра) — применяется автоматически.
- **Однозначное частичное совпадение** (одна сторона содержит другую как
  подстроку, ровно один кандидат) — применяется автоматически.
- **Совпадение по пересечению слов** (например, "Наталия и Нина" в таблице ~
  "Наталия Богданова и Нина" в базе — слово "Богданова" не даёт им совпасть
  подстрокой) — **никогда не применяется автоматически**, только как
  подсказка в отчёте `dry-run`. Строка остаётся непроимпортированной
  (`skip_client_unmatched`) до ручного решения.
- **Совпадений нет** — строка не импортируется, попадает в отчёт.

## Нормализация исполнителей

Ключ дедупликации — имя без "ё" (заменяется на "е"), в нижнем регистре, с
одним пробелом между словами. "Иван Тесёлкин" и "Иван Теселкин" схлопываются
в один `EditorProfile`; отображаемое имя — вариант **с** "ё", если он
встретился хотя бы раз (более корректное написание).

## Команды

```bash
set -a && source .env.local && set +a

npx tsx scripts/import-montage-projects/dry-run.ts     # ничего не меняет, только отчёт
npx tsx scripts/import-montage-projects/apply.ts        # создаёт монтажёров и проекты, пишет манифест
npx tsx scripts/import-montage-projects/rollback.ts scripts/import-montage-projects/backups/apply-....json
```

## Идемпотентность

Каждый `MontageProject` получает `importExternalId` — sha256 от
`дата|заказчик|название|исполнитель|сумма клиента|дедлайн` (не одно только
имя клиента, как явно требовало ТЗ). `buildPlan()` всегда проверяет, какие
`importExternalId` уже есть в базе, и помечает такие строки
`skip_already_imported` — повторный запуск `apply.ts` не создаёт дублей
проектов. Монтажёры сопоставляются по нормализованному имени с уже
существующими `EditorProfile` перед созданием — повторный запуск тоже не
плодит вторые профили.

## Откат

`apply.ts` пишет манифест в
`scripts/import-montage-projects/backups/apply-<дата>.json` со списком
реально созданных `{sheetRow, montageProjectId, fingerprint}`. `rollback.ts`
с этим файлом удаляет ровно эти проекты монтажа. **Монтажёров (`EditorProfile`)
откат не удаляет** — между `apply` и откатом администратор мог уже вручную
дополнить профиль (телефон, специализация), и откат не должен стирать эту
правку; профиль без единого проекта не мешает и нигде не считается ошибкой.
