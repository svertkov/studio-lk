# Резервное копирование базы данных (PostgreSQL)

Платформа 2470 использует управляемый PostgreSQL (Yandex Cloud Managed PostgreSQL, датасорс в `prisma/schema.prisma`). Эти скрипты создают полный дамп базы (`pg_dump`), сжимают его, считают контрольную сумму SHA-256, ведут журнал и удаляют копии старше срока хранения.

## Установка pg_dump (один раз)

Скриптам нужны системные утилиты `pg_dump`/`psql` (входят в клиент PostgreSQL) — сами по себе они не npm-пакет.

**На macOS** (через [Homebrew](https://brew.sh)):
```
brew install libpq
echo 'export PATH="/opt/homebrew/opt/libpq/bin:$PATH"' >> ~/.zshrc
source ~/.zshrc
pg_dump --version
```
(на Intel Mac путь будет `/usr/local/opt/libpq/bin`)

**В GitHub Actions** (`ubuntu-latest`) утилита уже предустановлена, либо ставится строкой `sudo apt-get install -y postgresql-client`.

## Переменные окружения

| Переменная | Обязательна | Назначение |
|---|---|---|
| `DATABASE_URL` | да (уже есть в `.env.local`) | продовая база — источник для `db:backup` и `db:restore:prod` |
| `BACKUP_DIR` | нет | куда складывать файлы. По умолчанию `~/studio-lk-backups` — **вне папки проекта**, не коммитится в git |
| `BACKUP_RETENTION_MONTHS` | нет | сколько месячных копий хранить, по умолчанию `12` |
| `TEST_DATABASE_URL` | да, для `db:restore:test` | отдельная тестовая база; должна отличаться от `DATABASE_URL` — скрипт откажется работать, если они совпадают |

## Команды

```bash
# Загрузить переменные окружения из .env.local в текущую сессию терминала
set -a && source .env.local && set +a

npm run db:backup                 # создать backup сейчас
npm run db:backup:list            # список сохранённых копий с checksum
npm run db:backup:verify -- studio2470-db-2026-07-01-0300.sql.gz     # проверить целостность файла
npm run db:restore:test -- studio2470-db-2026-07-01-0300.sql.gz      # восстановить в TEST_DATABASE_URL
npm run db:restore:prod -- studio2470-db-2026-07-01-0300.sql.gz --yes-i-understand-this-overwrites-production
                                   # восстановить В PRODUCTION — требует флаг
                                   # И ручной ввод фразы подтверждения в терминале,
                                   # не запускается из cron/CI по конструкции
```

## Что делает `db:backup`

1. `pg_dump --format=plain --no-owner --no-privileges` через переменные окружения `PGHOST/PGUSER/PGPASSWORD/...` (не через argv — так пароль не попадает в список процессов `ps aux`).
2. Сжимает поток в `.sql.gz`, пишет во временный файл, переименовывает в финальное имя только после успеха.
3. Отказывается сохранять пустой (0 байт) результат.
4. Считает SHA-256, сохраняет рядом как `<файл>.sha256`.
5. Ставит права `600` на файл и `700` на директорию (только владелец процесса).
6. Пишет запись в `backup.log` внутри `BACKUP_DIR` (даты, размер, checksum, ошибки — без пароля, он всегда маскируется перед логированием).
7. Удаляет копии старше `BACKUP_RETENTION_MONTHS`, но **никогда** не удаляет самый свежий успешный backup и ничего не удаляет, если сам новый backup не создался.

Имя файла: `studio2470-db-{YYYY-MM-DD}-{HHmm}.sql.gz`, время — по часовому поясу студии (`Europe/Moscow`), не по TZ машины, где запущен скрипт.

## Расписание

Эти скрипты сами по себе НЕ планировщик — они просто выполняют разовое действие при запуске. Способ регулярного (ежемесячного) запуска — открытый вопрос, зависит от того, где физически исполняется job (ноутбук не включён 24/7 → local cron/launchd не гарантирует срабатывание; альтернатива — GitHub Actions на существующем remote репозитория). См. заметку в памяти проекта — решение отложено по просьбе владельца.

## Известное ограничение окружения разработки

В песочнице, где выполнялась эта задача, нет установленного `pg_dump`/`psql` и нет прав на установку пакетов (`brew`/`docker` недоступны) — поэтому реальный end-to-end прогон `db:backup` + `db:restore:test` в этой сессии выполнить не удалось. Логика проверена: юнит-тестами (см. `scripts/db-backup/core.test.ts`) для чистых функций (имя файла, retention, парсинг DATABASE_URL, санитайзер логов) и вручную прочитанным кодом. Перед боевым использованием нужно один раз прогнать `npm run db:backup` и `npm run db:restore:test` на машине с установленным `pg_dump`/`psql` (см. раздел "Установка pg_dump" выше) и убедиться, что тестовое восстановление проходит успешно.
