# Единый источник данных и взаимосвязанность платформы

Studio LK — не набор независимых экранов, а одна система, в которой один и тот же заказ/клиент/оплата/абонемент виден из дашборда, CRM, «Заказов», расписания, карточки клиента, финансов и отчётов одновременно. Расхождение данных между этими экранами (пример: раздел «Заказы» показывал «Нет данных» по оплате, хотя она уже была заполнена в карточке записи — исправлено 2026-07-12, см. `src/lib/payment-model.ts`) — это баг архитектуры, а не косметика одной таблицы, и такие баги повторяются, если для каждого экрана писать свою версию одной и той же логики.

Обязательные правила при любой будущей задаче:

1. Перед добавлением нового функционала сначала проверь, не существуют ли уже эти данные в другом разделе платформы (Prisma-схема, `src/lib/actions/*`, существующие DTO) — не создавай их заново «для удобства одного экрана».
2. Одни и те же данные не хранятся отдельными копиями для разных экранов (никаких `orderListPayment`, `dashboardPayment`, `clientOrderPaymentCopy` и т.п.). Если у сущности уже есть каноническое поле — новый экран читает его, а не заводит своё.
3. Если у заказа есть связанная запись в расписании (`ScheduleEvent`), её поля — источник правды поверх собственных полей `Order` (тот же принцип двойного источника, что уже применяется для `comment`/`notes`, `promotionType` и стоимости/оплаты — см. `toDTO()` в `src/lib/actions/orders.ts`). Не изобретай новый способ решать, «чьё поле главнее» — расширяй этот же принцип.
4. Повторяющаяся логика чтения/интерпретации данных живёт в ОДНОМ общем helper/service/selector (например `src/lib/payment-model.ts`, `src/lib/promotion-model.ts`), а не копируется по компонентам. Если увидел вторую копию одной и той же логики — это повод вынести её, а не размножить третий раз. Пример объединения нескольких независимых источников в одну агрегирующую функцию, которую читают несколько KPI-карточек сразу, не пересчитывая каждая своё: `computeCombinedFinanceSummary` (`src/lib/finance-model.ts`) — собирает выручку/расходы/прибыль страницы «Финансы» из визитов (`ClientVisit`), обязательств (`Expense`) и монтажа (`computeMontageDashboardStats`), не суммируя их заново в `page.tsx`.
5. Изменение данных в одном разделе обязано автоматически отражаться во всех связанных — через `revalidatePath` на все затронутые маршруты (см. `revalidateOrderPaths()` в `src/lib/actions/orders.ts` и инвалидацию в `upsertScheduleEvent`, `src/lib/actions/schedule.ts`), а не через ручную перезагрузку страницы администратором.
6. Перед завершением задачи перечисли явно, какие экраны читают/пишут затронутые данные (обычно это часть или все: дашборд, карточка заказа, «Заказы», CRM, расписание, карточка клиента, финансы, абонементы, монтаж, материалы, отчёты) и проверь каждый из них — глазами (Playwright) или явным рассуждением, если браузерная проверка недоступна.
7. При создании нового поля в схеме — то же самое: пройдись по списку разделов из пункта 6 и реши, кто из них должен его видеть/менять, прежде чем писать код.
8. Не создавай отдельное поле «только чтобы показать на одной странице» — если данные нужны для отображения, они должны браться из канонического источника (при необходимости — через новый чистый helper), а не дублироваться.
9. После любой мутации (create/update/delete любой связанной сущности) — инвалидируй/обнови все связанные кеши и маршруты, а не только тот экран, с которого вызвана мутация.
10. Задача считается завершённой только после проверки сквозного сценария: данные, изменённые в одном разделе, видны в неизменном виде во всех остальных, где они отображаются.

## Чеклист для каждой новой задачи, трогающей данные заказа/клиента/оплаты/абонемента

- Где хранится источник истины (какая модель/поле в `prisma/schema.prisma`)?
- Какие разделы читают эти данные?
- Какие разделы могут их изменять?
- Как выполняется синхронизация между ними (дуал-сорсинг при чтении? зеркальная запись? оба)?
- Какие пути инвалидируются (`revalidatePath`) после мутации?
- Не создаются ли дублирующие поля/копии ради одного экрана?
- Обновлены ли тесты всех затронутых чистых helper-функций?
