# Единый источник данных и взаимосвязанность платформы

Studio LK — не набор независимых экранов, а одна система, в которой один и тот же заказ/клиент/оплата/абонемент виден из дашборда, CRM, «Заказов», расписания, карточки клиента, финансов и отчётов одновременно. Расхождение данных между этими экранами (пример: раздел «Заказы» показывал «Нет данных» по оплате, хотя она уже была заполнена в карточке записи — исправлено 2026-07-12, см. `src/lib/payment-model.ts`) — это баг архитектуры, а не косметика одной таблицы, и такие баги повторяются, если для каждого экрана писать свою версию одной и той же логики.

Обязательные правила при любой будущей задаче:

1. Перед добавлением нового функционала сначала проверь, не существуют ли уже эти данные в другом разделе платформы (Prisma-схема, `src/lib/actions/*`, существующие DTO) — не создавай их заново «для удобства одного экрана».
2. Одни и те же данные не хранятся отдельными копиями для разных экранов (никаких `orderListPayment`, `dashboardPayment`, `clientOrderPaymentCopy` и т.п.). Если у сущности уже есть каноническое поле — новый экран читает его, а не заводит своё.
3. Если у заказа есть связанная запись в расписании (`ScheduleEvent`), её поля — источник правды поверх собственных полей `Order` (тот же принцип двойного источника, что уже применяется для `comment`/`notes`, `promotionType` и стоимости/оплаты — см. `toDTO()` в `src/lib/actions/orders.ts`). Не изобретай новый способ решать, «чьё поле главнее» — расширяй этот же принцип.
4. Повторяющаяся логика чтения/интерпретации данных живёт в ОДНОМ общем helper/service/selector (например `src/lib/payment-model.ts`, `src/lib/promotion-model.ts`), а не копируется по компонентам. Если увидел вторую копию одной и той же логики — это повод вынести её, а не размножить третий раз. Пример объединения нескольких независимых источников в одну агрегирующую функцию, которую читают несколько KPI-карточек сразу, не пересчитывая каждая своё: `computeCombinedFinanceSummary` (`src/lib/finance-model.ts`) — собирает выручку/расходы/прибыль страницы «Финансы» из визитов (`ClientVisit`), обязательств (`Expense`) и монтажа (`computeMontageDashboardStats`), не суммируя их заново в `page.tsx`.
5. Изменение данных в одном разделе обязано автоматически отражаться во всех связанных — через `revalidatePath` на все затронутые маршруты (см. `revalidateOrderPaths()` в `src/lib/actions/orders.ts` и инвалидацию в `upsertScheduleEvent`, `src/lib/actions/schedule.ts`), а не через ручную перезагрузку страницы администратором.
6. Перед завершением задачи перечисли явно, какие экраны читают/пишут затронутые данные (обычно это часть или все: дашборд, карточка заказа, «Заказы», CRM, расписание, карточка клиента, финансы, абонементы, монтаж, материалы, отчёты) и проверь каждый из них — глазами (Playwright) или явным рассуждением, если браузерная проверка недоступна.
7. При создании нового поля в схеме — то же самое: пройдись по списку разделов из пункта 6 и реши, кто из них должен его видеть/менять, прежде чем писать код.
8. Не создавай отдельное поле «только чтобы показать на одной странице» — если данные нужны для отображения, они должны браться из канонического источника (при необходимости — через новый чистый helper), а не дублироваться.
9. После любой мутации (create/update/delete любой связанной сущности) — инвалидируй/обнови все связанные кеши и маршруты, а не только тот экран, с которого вызвана мутация.
10. Задача считается завершённой только после проверки сквозного сценария: данные, изменённые в одном разделе, видны в неизменном виде во всех остальных, где они отображаются.

**Прецедент нарушения правил 1/4 (2026-07-19):** при добавлении нового значения `DocumentFlowType` (`CONTRACT_APPENDIX_INVOICE_ACT`) обнаружилось, что состав требований по режиму документооборота («какие типы флоу требуют счёт/акт») был закопирован в ТРИ независимых места: `document-model.ts` (не экспортированные `FLOW_TYPES_REQUIRING_ACT`/`_INVOICE`), инлайн-условие внутри `getDocumentsDashboardStats` (`src/lib/actions/documents.ts`), и отдельный локальный `string[]`-список прямо в `OrdersListView.tsx`. Добавление нового значения enum потребовало бы правки во всех трёх местах по отдельности — если забыть хоть одно, новый режим стал бы вести себя по-разному в разных разделах платформы без единой ошибки компиляции, потому что списки были типизированы как `string[]`, а не `DocumentFlowType[]`. Исправлено: константы экспортированы из `document-model.ts`, оба остальных места теперь читают их напрямую. **Практический вывод:** при добавлении нового значения в существующий enum — не полагаться на память/поиск по одному файлу, а сделать `grep -rn` по имени enum и по каждому известному его значению (`grep -rn "CONTRACT_INVOICE_ACT"` и т.п.) по всему `src/`, прежде чем считать функционал добавленным во все места. Если хотя бы один найденный список/условие типизирован как `string[]`/`string`, а не как сам enum-тип — это сам по себе сигнал дублирующей копии, которую стоит вынести в общий helper (см. правило 4 выше), а не оставить как есть.

## Редактируемость связанных сущностей

2026-07-22: карточка заказа полгода показывала приложение к договору только для чтения — не было формы редактирования номера/даты/суммы после создания. Не отдельная теория, а прямое следствие того, что при первом добавлении сущности в UI (документ, приложение, любая другая связанная запись) редактирование откладывалось «на потом» и потом не делалось. Правила:

1. Любая отображаемая связанная сущность (документ, приложение, проект монтажа, счёт и т.п.) обязана иметь понятный путь редактирования из того места, где она показана — не только создание и просмотр.
2. Нельзя создавать документ/запись, которую потом невозможно изменить из рабочего контекста, где администратор её видит — не откладывать редактирование «на будущую доработку» по умолчанию.
3. Редактирование связанной сущности переиспользует ОДНУ форму/компонент независимо от того, из скольких карточек к ней есть доступ (см. `AppendixEditDialog.tsx` — один компонент, встроенный в общий `WorkDocumentsSection.tsx`, а не по копии на каждую карточку).
4. Если редактор связанной сущности открывается поверх большой формы с автосохранением (см. `use-autosave.ts`) — перед открытием обязателен `flush()` черновика родительской формы (см. `onBeforeEditRelated` в `WorkDocumentsSection.tsx`), чтобы не потерять несохранённые данные основной карточки.
5. Изменение номера/реквизита документа обязательно проверяется на сервере (уникальность, права), а не только валидацией в форме — скрытие кнопки в UI это не защита сама по себе.
6. Изменение документных реквизитов — значимое действие, пишется в audit log отдельным событием (см. `APPENDIX_NUMBER_CHANGED`), а не молча растворяется в общем `_UPDATED`.
7. После изменения связанной сущности обновляются ВСЕ разделы, где она отображается (см. правило 5/9 «Единого источника данных» выше) — включая клиента, если сущность привязана к клиенту не напрямую, а через промежуточную запись (приложение → договор → клиент, см. `revalidateDocumentPaths` в `updateDocument`).
8. Запрещено хранить копию номера документа отдельно в заказе/CRM/клиенте/финансах — все экраны читают одно поле сущности (см. `getDocumentDisplayNumber`).

## Data Safety and Audit Integrity

2026-07-15: одноразовый ad-hoc Node/Prisma-скрипт (написан и запущен Claude Code для уборки одной тестовой строки) обратился к несуществующему полю (`order.scheduleEventId` вместо `order.scheduleEvent.id`), получил `undefined`, и `prisma.auditLog.deleteMany({ where: { entityId: undefined } })` превратился в удаление БЕЗ условия — стёрло все 1799 строк журнала аудита (`cms_audit_log`) в единственной продовой базе. См. память `project_studio_lk_audit_log_incident`. Правила ниже — прямая реакция на это, не общая теория; жёстко привязаны к реальному масштабу платформы (один владелец, одна база, Yandex Managed PostgreSQL, деплой через Vercel, без CI/Docker) — не разрастаться до enterprise-архитектуры (hash-цепочек, outbox pattern, множества окружений), которая этому масштабу не соответствует.

1. **`cms_audit_log` физически append-only на уровне БД**, не только по соглашению в коде: `BEFORE UPDATE/DELETE/TRUNCATE` триггер (`prisma/manual-sql/audit-log-append-only.sql`) отклоняет исключением любую попытку изменить/удалить существующую строку — для ЛЮБОЙ роли, включая ту, которой работает всё приложение. Не управляется Prisma (`db push`/`migrate` не знает о триггерах) — применяется вручную через `psql -f`, см. комментарий в самом файле. Не добавляй в код методы вроде `deleteAuditLog`/`clearAuditLog`/`purgeAuditLog` — их и не должно быть, БД всё равно откажет.
2. **Перед ЛЮБЫМ `deleteMany`/`updateMany`/`delete` в ad-hoc скрипте против продовой базы** — сначала `findFirst`/`findUnique`, чтобы увидеть реальную строку и её `id`, и удалять/менять по этому найденному литералу, а не по повторно вычисленной переменной. Prisma молча трактует `undefined` в `where` как «условие отсутствует» — это и есть ровно тот механизм, который стёр таблицу целиком. Если значение в `where` пришло не литералом, а через переменную/поле объекта — залогируй его перед выполнением и убедись, что оно не `undefined`.
3. **Критичная бизнес-история не должна зависеть только от общего `AuditLog`.** Если для сущности требуется предъявляемая пользователю история изменений (пример: `SubscriptionAdjustment` — история корректировок часов абонемента, `prisma/schema.prisma`) — заведи для неё отдельную иммутабельную таблицу, и пиши в неё в ОДНОЙ транзакции вместе с самим изменением данных и записью в `AuditLog` (см. `updateSubscriptionHours`/`chargeEventToSubscription`, `src/lib/actions/subscriptions.ts`) — не должно быть ситуации «данные изменились, история — нет» из-за падения между двумя отдельными вызовами.
4. **Если специализированная история не восстановима за период до какой-то даты** (как случилось здесь) — UI обязан честно показать это отдельной пометкой («История до … частично недоступна из-за ранее произошедшей потери журнала»), а не молчаливым «корректировок не было» (см. `getSubscriptionAdjustmentHistory` + `SubscriptionDetailModal.tsx` как образец). Не выдавай реконструированные/неполные данные за полную историю.
5. **Ежедневный backup БД настроен через GitHub Actions** (`.github/workflows/db-backup.yml`, cron 00:00 UTC = 03:00 МСК), не зависит от того, включён ли ноутбук. Секрет `DATABASE_URL` уже в репозитории (`gh secret list`). Артефакт хранится 90 дней (лимит GitHub) — это не полноценный долгосрочный архив (12+ месяцев), известное ограничение, не решённое в этом проходе. **Не считай "настроен" равным "работает"** — 2026-07-22 обнаружилось, что все запланированные бэкапы молча падали как минимум 6 дней подряд (`pg_dump: error: aborting because of server version mismatch` — apt даёт postgresql-client 16, управляемый Postgres в Yandex Cloud уже 17.x). Перед любой операцией, для которой нужен свежий backup, проверь `gh run list --workflow=db-backup.yml`, а не полагайся на то, что job существует.
6. **Перед любой деструктивной операцией** (миграция, меняющая/удаляющая колонку, массовое изменение данных, восстановление backup) — сначала `npm run db:backup` вручную (или ручной запуск `gh workflow run db-backup.yml`, если `pg_dump` не установлен локально), если с последнего автоматического бэкапа прошло значимое время, и проверка, что `DATABASE_URL` действительно указывает на ожидаемую базу (`psql "$DATABASE_URL" -c "SELECT current_database();"`). Перед миграцией, меняющей ТИП существующей колонки (как `Document.number` Int→String, 2026-07-22) — сначала прочитать реальные данные в затронутых колонках (сколько строк, какие значения), чтобы оценить риск заранее, а не полагаться только на то, что Prisma не попросит `--accept-data-loss`.
7. **Никогда не выполнять `prisma migrate reset`, `DROP DATABASE`, `TRUNCATE` на продовой базе** — единственная база в проекте одновременно и dev, и prod (осознанное решение владельца, не ошибка). Любая команда, которая в обычном проекте была бы «сбросить dev-базу», здесь равна «стереть все рабочие данные платформы».
8. **DB-роль приложения (`studio_lk_user`) — владелец всех таблиц** (нет отдельной роли с урезанными правами для рантайма) — осознанно отложено (см. память), не единственная защита. Не полагайся только на права роли — полагайся на append-only триггер (п.1) и дисциплину при ad-hoc скриптах (п.2).
9. **Любая потеря/подозрение на потерю данных** — сначала честный отчёт (что удалено, когда, почему, что пострадало в интерфейсе, что доступно для восстановления), потом код. Не говорить «всё работает», если исторические данные утрачены — явно перечислить, что именно и как это видно пользователю.

## Реестр документов (договоры/счета/акты)

Платформа ведёт структурированный реестр номеров и статусов документов (`Document`, `src/lib/document-model.ts` + `src/lib/actions/documents.ts`) — **не файловое хранилище**. PDF/файлы не хранятся и не должны храниться без отдельного нового решения пользователя; наличие `Document` не означает существование файла.

1. Номер комплекта работы (`Счёт №127-1`, `Акт №127`) живёт на самой работе — `Order.documentPackageNumber` / `MontageProject.documentPackageNumber`, а не копируется в каждый `Document`. Отображаемый номер — чистая функция `getDocumentDisplayNumber`, не хранимое поле.
2. Статус оплаты счёта НЕ хранится на `Document` — читается из существующего источника (`Order.paymentStatus` через `payment-model.ts` для заказов, `MontageProject.clientPaymentStatus` для монтажа), через `getDocumentPaymentState`. Не заводить второй параллельный `paymentStatus` на документе.
3. Сквозные номера (договор, комплект работы) выдаются только через `DocumentCounter` (атомарный upsert+increment) — не генерировать номера иначе, не присваивать вручную без прохождения через `bumpCounterIfLower`.
4. RBAC для документов пока не реализован (как и почти везде в проекте) — не предполагать, что он есть, при расширении этой области.
5. Новый документный функционал по-прежнему проверяется по общему чек-листу ниже (клиент, заказ, монтаж, CRM, финансы, дашборд, документы).
6. `APPENDIX` (приложение к договору) — номер сквозной **в рамках одного договора** (`contractId`+`number`, `getNextAppendixNumber`), НЕ через `DocumentCounter` — в отличие от номера самого договора и номера комплекта работы. Клиент приложения не хранится напрямую (`clientId` остаётся null) — определяется через `contract.clientId` (см. ветку `{ type: 'APPENDIX', contract: { clientId } }` в `getDocumentsForClient`).
7. `serviceDescription` (описание услуги) — ручной свободный текст у `APPENDIX`/`INVOICE`/`ACT`, независимый у каждого документа. Кнопки «Заполнить из…» в UI — одноразовое копирование значения в момент создания, НЕ живая ссылка/синхронизация: изменение текста в одном документе не должно менять текст в другом задним числом.
8. Модель осознанно спроектирована так, чтобы в будущем можно было добавить хранение файла/ЭДО/автогенерацию без изменения текущих полей — не переиспользовать существующие поля не по назначению ради будущей интеграции сейчас, для этого заводятся новые nullable-поля поверх.
9. Timeline документооборота (создан → приложение → счёт → оплата → акт → закрыт) не реализован — задел на будущее, не делать вид, что он есть.
10. Суммы приложений используются только для сверки/предупреждения (`appendixAmountMismatches`, `MontageFinanceBreakdown.tsx`) — источник финансовой истины остаётся `Order.preliminaryAmount`/`MontageProject.clientAmount`, ничего не пересчитывается автоматически.
11. Историческая AI-миграция данных из Google-таблиц (см. отдельную задачу) не переносит приложения — в исходных таблицах студии их нет вообще, только договоры/счета/акты.

## Булевы бизнес-исключения — не голый checkbox

Булевы поля, представляющие **осознанное бизнес-исключение** (не «настройка интерфейса» и не «фильтр списка»), должны реализовываться через `ConfirmableStatusToggle` (`src/components/ui/confirmable-status-toggle.tsx`), а не голым `<input type="checkbox">`:

1. Обычные фильтры списков — `ToggleChip` (`src/components/ui/toggle-chip.tsx`), без диалога подтверждения.
2. Безопасные настройки интерфейса (не влияющие на бизнес-логику/отчётность) — обычный переключатель/чекбокс.
3. Значимое бизнес-исключение (пример: «Ссылка на Яндекс.Диск не требуется», `ScheduleEvent.yandexLinkRequired`/`nasLinkRequired`) — только `ConfirmableStatusToggle`: подтверждение диалогом на включении, аудит через `writeAuditLog`, и рядом с самим флагом — `confirmedAt`/`confirmedById`/`reason` (см. `yandexNotRequiredConfirmedAt/By/Reason` на `ScheduleEvent` как образец — метаданные живут на той же записи, не в отдельной таблице).
4. Подтверждённое исключение не должно маскироваться под «поле забыли заполнить» — визуально это отдельное состояние (предупреждающий контур, disabled-поле, поясняющий текст «Подтверждено: …»), а не пустое поле как обычно.
5. Все разделы, показывающие это поле (дашборд, заказ, карточка клиента, таблица «Заказы», отчёты), должны читать одно и то же состояние — никаких по-разному интерпретирующих его копий.
6. Любая новая проверка «заполнено/не хватает» обязана различать три исхода: заполнено / пусто-и-обязательно / пусто-но-подтверждённо-не-требуется — не сводить последние два в один.
7. Возврат из исключения в обычный режим (`false → true`) не требует подтверждения — это возврат к более строгой проверке, а не новое исключение.
8. Перед добавлением нового чекбокса — сначала определить, к какой из трёх категорий выше он относится (см. `RequiredLinkToggle` → `ConfirmableStatusToggle`, 2026-07, как прецедент).

- Какие разделы могут их изменять?
- Как выполняется синхронизация между ними (дуал-сорсинг при чтении? зеркальная запись? оба)?
- Какие пути инвалидируются (`revalidatePath`) после мутации?
- Не создаются ли дублирующие поля/копии ради одного экрана?
- Обновлены ли тесты всех затронутых чистых helper-функций?
